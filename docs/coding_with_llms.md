# Coding with LLMs

Coding with LLMs has come a long way since 2023 when I first started with them. This document captures a little about that journey, the current state(08-14-2025) and my current workflow. This will begin in narrative form and I will redux at the end and move it into a ;tldr format.

--During the first two weeks of working with LLMs, building a text-to-sql demo app, I saw some of the fundamental challenges working with LLMs: non-deterministic responses, high sensitivity to minimal context differences, dynamic workflow based on responses.--

I realized very quickly when working with LLMs that I would have to be very cognizant of my language and learn to formulate my questions to get the desired response. Using prompts I could nudge the LLM but ultimately it was more my responsibility to understand how to craft a question or instruction than the models to "understand" my words. I was excited about this because software development is primarily a task of enginerring to some specification. I have always balked at the ubiquitoius selling point of most new "agentic" tools that claim they will create a whole application from just one or two sentences. That is not understanding intent, just fancy copy and paste.

There are multiple industries dedicated to agile methods, user stories and personas, unit and integration testing, all of which try and help to define how to build software from business reqs to prod devops. 


## Main Problems with LLMs
    1. LLM's almost always say and do too much. A good starting prompt for everything could be "say less". Eagerness -> confabulation
    2. Lots of other tools are needed to gather and format all the required information just to start like 3rd party docs.


## Copilot based SDLC
    1. Exploration, documentation and prototyping
       1. The goal of this first step is to gather all the information you need to clearly define the goals, features and architecture
       2. Create feature sets based on v0.0.1, v0.1.0, v0.2.0
    2. v0.X.X development
       1. Specification and test
       2. Realistic data and examples for integration testing
       3. No unit testing required
       4. Update documentation with every release
    3. Evaluate and refactor v0.X.X
